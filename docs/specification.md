# 仕様書 (Specification)

個人用TODO CLI「my-todo」の機能仕様を定義する。本書は `README.md` のラフな構想を整理し、スコープを明確化したもの。実装レベルの設計は [`design.md`](./design.md) を参照。

## 1. 目的・背景

個人開発初心者でも安心して使える、**自分の欲しい機能だけ**を備えた最小TODO。
世の中のTODOアプリは多いが、初心者向けは作者・更新状況の見極めが面倒で、また人によって必要な機能が異なり、過剰な機能はかえって認知負荷になる。そのため必要な部分だけ自作する。

設計の通底する思想:
- **認知負荷を増やさない**。自動で収集・管理できる情報はユーザーの意識から外す。
- **おおまかな寿命の把握**ができればよい。きっちりした期限管理はしない。

## 2. スコープ

### 2.1 やること（欲しい機能）

| ID | 機能 | 内容 |
|----|------|------|
| F-1 | グローバルCLI | CLIから起動でき、**どのパスから実行しても同一のストレージにアクセス**できる |
| F-2 | 日付の自動記録 | 作成日・更新日を自動で記録する。自動収集できる項目はユーザーが意識しない |
| F-3 | vim編集 | タスク本文の編集は `$EDITOR`（vim）で開く |
| F-4 | to think リスト | 思考が長くなりそうな項目を、通常タスクとは**別立て**の「to think」リストとして管理する |
| F-5 | ライフサイクル分類 | タスクを寿命（短期 / 中期 / 長期）で分類し、長期間未解決のものを**徐々に長寿命側へ自動的に移行**する |
| F-6 | 一覧表示 | タスクの一覧を表示する |

### 2.2 やらないこと（いらない機能）

- **構造化**（ネスト・サブタスク階層など）
- **通知**
- **きっちりした期限管理**（厳密な締切があるものはカレンダー側で管理する。本アプリはおおまかな寿命把握のみ）
- **高度な検索**

### 2.3 将来検討（微妙な機能）

- **タグ**
  - タグベースの管理は基本的に**やりたくない**（原則として採用しない方針）。
  - ただし長寿命タスクが残り続ける場合、「タスクとして残す」より「タグへ変換し、小タスク群のみを残す」方が総タスク数を減らせるケースがある。
  - 例: 「xxxの発光過程を見る」（長期間残っているタスク）を
    1. xxxの最適化
    2. 既存実験のxxx発光スペクトル収集
    3. 最適化構造の一点計算 …
    に分解し、元タスクを削除。代わりに各小タスクに「xxx発光」タグを付ける。
  - この**分解→タグ変換ワークフロー**に限り将来検討の余地あり。詳細な設計は今回スコープ外。

## 3. 利用イメージ（概略）

- ターミナルから `todo add "..."` で短期タスクを追加（作成日は自動記録）。
- `todo ls` でライフサイクル段階ごとに一覧を確認。
- 思考が必要な項目は to think リストへ。
- 何もしなくても、古くなったタスクは中期→長期へと自動で移っていき、「おおまかな寿命」が一覧から把握できる。

具体的なコマンド体系・データ構造・自動移行の仕組みは [`design.md`](./design.md) を参照。
