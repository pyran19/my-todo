## なぜ作るのか

個人開発初心者御用達のTODOリスト。
腐るほど世にあるので今更自作するのも死ぬほどばからしいんだけど、初心者御用達のせいで変な人の変なアプリ踏まないようにするのもだるいし
更新止まってないか確認するのもだるい。人によってほしい機能も変わるし過剰な機能はそれはそれで認知の負荷。
自分の欲しい部分だけ自作するのが無難。

## 欲しい機能

- cliから開いて、どのパスから実行しても同じストレージにアクセスできる
- 日付の自動記録. (自動収集できる箇所は意識から外す)
- エディタはvim
- to think. 長くなりそうな項目をto thinkリストとして別立て。
- ライフサイクルごとの分類。寿命ごとに複数テーブル用意して、長期間未解決のものは徐々に長寿命用のテーブルに移していく
- 一覧

## いらない機能

- 構造化
- 通知
- 期限(きっちりとした期限あるものは別途カレンダーで管理してるので、おおまかな寿命の把握分のみ)
- 高度な検索

## 微妙（採用を迷っていた）機能

- タグ
  - タグベースの管理とか絶対にやりたくない
  - が、長寿命のタスクが残ってる場合タスクとして残すよりタグと言う形に変換して小タスクのみを残す+小タスクにタグを付ける方がタスク減る
  - c.f. 英語を身につける(長期間残ってるタスク)→ 1. 単語帳を1周する, 2. 文法書を読み終える, 3. 英語の記事を1本読む,...に分解して 英語を身につけるタスクを消す。代わりにそれぞれのタスクに英語のタグ

## 技術詳細（当初の想定）

- uvのtoolインストールでグローバルに使えるようにする
- ホーム下にアプリ用ディレクトリ作ってそこにデータ貯める。

## その後の整理メモ

当初の構想と、実装に落とし込む過程での主な判断の差分を記録しておく
（詳細な根拠は `design.md` 側に残している）。

- 「寿命ごとに複数テーブルを用意して移す」案は、単一 `tasks` テーブル + `lifecycle`
  カラムに整理した。テーブル間のレコード移動より、カラム値の更新の方が単純で同じ意図を満たせるため。
- 「ホーム下にアプリ用ディレクトリ」は、XDG Base Directory 準拠の
  `~/.local/share/my-todo/` として具体化した。
- タグは構想どおり今回は未実装。将来の「分解→タグ変換」ワークフローのときに再検討する。
